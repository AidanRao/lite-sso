# 代码规范

## 包设计与管理
【原则】：单一职责

【建议】1.1 以package（包）的形式组织代码，每个package有若干.go文件组成，每个go文件内部应依次按照如下顺序进行组织：
- 包声明
- 包导入
- 常量定义
- 全局变量定义
- 类型定义
- 方法/函数定义  (package级别方法在前，类型方法在后，公有方法在前，私有方法在后)

【规则】1.2 import规范应该遵循：
- import包时不允许使用点.操作，测试除外
- import包时不允许使用别名，同时导入2个相同的包名除外
- import 导入包时统一使用小括号，包名要另起一行

## 命名
【原则】见名知意，对于类型定义、变量、包、函数，接口等统一驼峰法
【说明】采用正确的或者通用的缩写来命名变量、函数、包


## 注释
【原则】对于注释，能够准确反映设计思想和代码逻辑，描述业务含义，使得其他人能够快速了解代码背景。一个模块中，统一使用中文或者英文，避免中英混搭；避免“中式英语”。

【建议】3.1  Go支持  /* Comments */ 注释以及 // Comments 注释。 通常建议采用 // Comments 注释，只有大段注释或者package 头部版权声明

【建议】3.2 对于可以导出的名称，建议都加上注释，这样在使用godoc的时候，可以清晰地生成项目文档，同时修改代码的同时，对注释进行相应的修改

【建议】3.3 每个包都应该包含注释，能够为package进行必要的说明

【规则】3.4 注释内容与// 留有空格，且避免语法错误的英文注释OR中式英语，使用全中文或者正确的英文进行注释


## 初始化与声明

【规则】4.1 init函数每个文件最多只有一个，每个包只有一个
【说明】不应该依赖于Go本身的init机制，散乱的init对新人非常容易误会。显式的初始化某些模块更容易让开发者知道自己正在做的事情。

【规则】4.2 对于struct的变量初始化,必须加上struct的field

【规则】4.3 如果 struct 中的域需要进行 json 序列化，则需要以大写字母开头，同时要进行json命名
【说明】 结构体中的变量以大写字母开头，可以保证 json.Marshal 的时候数据持久化正确。如果结构体中的变量以小写字母开头，则使得 json.Marshal 的时候忽略该字段，使得该字段的值丢失。从而 json.Unmarshal 的时候将该变量的值置为默认值。如果不加json命名，序列化时会使用默认的大写方式。

【规则】4.4 bool类型变量直接应直接进行真假判断，避免与true，false比较

## 测试

【原则】AIR原则： A: Automatic(自动化) I：Independent(独立性) R:Repeatable(可重复)

【规则】5.1 遵守Go的单测规范，例如测试文件名以  待测试文件_test.go 命名， 测试方法以 Test_待测试方法_条件 命名
【建议】5.2 建议利用testify做断言

## 工程结构
默认上层依赖下层，箭头关系表示可以直接依赖，例如开放接口可以依赖于web层，也可以直接依赖Service层

- 开放接口层：可以直接封装service暴露RPC接口；通过web层封装成HTTP接口
- 展示层：对接模板渲染并执行显示的层次（前端）
- Web层：对接HTTP访问，对各类http基本参数校验
- Service层：相对具体的业务逻辑处理层，整合DAL和通用处理层向上提供服务
- Manager、Handle层：通用业务处理层，有如下特征，a. 对第三方平台调用进行封装，预处理结果以及转化异常信息，b. 对Service层的通用能力下沉，例如缓存，中间件等处理 c.与DAL层交互，对多个DAO复用
- DAL层：数据访问层，与底层MySQL，Redis等交互
- 外部平台、第三方Proxy层：外部接口或第三方平台，包括其他服务的RPC接口，HTTP接口等
